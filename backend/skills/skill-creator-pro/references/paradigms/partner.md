# Partner（交互协作）

> 占比：14% — 人机协同的核心范式

## 定义

**核心职责**：Structure human-agent collaboration.

**适用场景**：
- 协同设计（与用户一起迭代方案）
- 分阶段审批（每步需要用户确认才继续）
- 迭代起草（草稿→反馈→修改→确认循环）
- 结构化访谈（通过提问收集信息，逐步构建产出）

Partner 的本质是对话协议设计者。它定义了 Agent 和人类之间的交互契约：什么时候 Agent 主动行动，什么时候必须等待人类确认，什么时候移交控制权。

## SKILL.md 结构指导

### 保留在 SKILL.md 中的内容

- **Opening Protocol**：启动时如何与用户建立协作契约
- **Confirmation Points**：哪些节点必须等待用户确认
- **Stage Exits**：每个协作阶段的退出条件

### 推入 `references/` 目录的内容

- 详细提示词模板（每阶段使用的提问模板）
- 示例对话（展示理想的协作流程）
- 输出格式参考

```
skill-name/
├── SKILL.md          # 开场协议 + 确认检查点 + 阶段退出条件
└── references/
    ├── prompts/      # 各阶段提示词模板
    ├── examples/     # 示例对话/输出
    └── formats/      # 输出格式参考
```

## 关键设计原则

### 1. 对话契约（Conversation Contract）

SKILL.md 必须定义清晰的对话契约，包括：
- Agent 的角色和行为边界
- 用户的期望参与方式
- 信息交换的格式和节奏

### 2. 人工检查点（Human Checkpoints）

在关键决策节点，Agent 必须暂停并等待用户确认。检查点的设置原则：
- 不可逆操作之前必须确认
- 方向性决策之前必须确认
- 大量工作开始之前必须确认

### 3. 渐进式确认（Progressive Confirmation）

不要一次性向用户倾倒所有信息。采用渐进式：先展示概要 → 等待反馈 → 再展开细节。每轮交互的信息量应该是用户可消化的。

### 4. 开场仪式（Opening Ritual）

Partner 必须有明确的开场协议，在协作开始时：
- 确认用户的目标和约束
- 说明协作流程和预期时间
- 设定沟通偏好（详细 vs 简洁、中文 vs 英文等）

## 反模式警告

### 1. 缺少确认检查点

**症状**：Agent 收到用户请求后一路执行到底，中间没有任何确认。
**后果**：方向错误时浪费大量工作，用户感觉失去控制。
**修复**：在每个关键决策节点插入确认检查点，至少包括：需求确认、方案确认、产出确认。

### 2. 即兴开始

**症状**：没有开场协议，Agent 直接开始工作，没有确认用户的真实需求。
**后果**：理解偏差导致返工，用户体验差。
**修复**：设计开场协议，至少包含 2-3 个澄清问题。

### 3. 一次性输出

**症状**：Agent 一次性输出完整结果，没有分阶段展示和迭代。
**后果**：用户面对大量信息难以消化，反馈也变得困难。
**修复**：将输出拆分为多个阶段，每阶段展示一部分，等待反馈后再继续。

## 典型代表

### brainstorming（头脑风暴协作者）

- 开场：确认头脑风暴的主题和约束
- 阶段 1: Agent 提出初始想法 → 用户反馈偏好
- 阶段 2: Agent 基于反馈深化 → 用户筛选
- 阶段 3: Agent 整理最终方案 → 用户确认
- 每阶段都有确认检查点，用户可随时调整方向

### doc-coauthoring（文档共创）

- 开场：确认文档类型、受众、篇幅
- 渐进式协作：大纲 → 逐章节展开 → 细节打磨
- 每个章节完成后等待用户审阅
- references/ 包含各类文档的格式模板

### plannotator（计划标注器）

- 开场：确认计划目标和时间范围
- 协作流程：粗略计划 → 标注风险点 → 细化时间线 → 确认里程碑
- 分阶段确认，确保计划符合用户的实际约束

## 与相邻范式的区别

### Partner vs Scout

| 维度 | Partner | Scout |
|------|---------|-------|
| 交互对象 | **人类**用户 | **环境**（文件系统、代码库） |
| 等待原因 | 等待用户确认/反馈 | 等待环境侦察结果 |
| 控制流 | 人类决定方向 | Agent 自主决定侦察路径 |
| 不确定性来源 | 用户需求不明确 | 环境状态未知 |

### Partner vs Orchestrator

| 维度 | Partner | Orchestrator |
|------|---------|-------------|
| 协调对象 | Agent + 人类 | Agent + 多工具/多 Agent |
| 交互模式 | 对话式 | 调度式 |
| 检查点 | 人工确认点 | 自动化交接点 |
| 人类参与度 | 高（每步都参与） | 低（只在关键节点参与） |
