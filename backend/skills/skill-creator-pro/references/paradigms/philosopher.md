# Philosopher（原则驱动）

> 占比：3% — 最稀有但最具影响力的范式

## 定义

**核心职责**：Establish principles that govern execution.

**适用场景**：
- 设计宪法（定义不可违反的规则体系）
- 推理框架（建立推理质量标准和验证方法）
- 治理规则（定义操作边界和行为约束）
- 元规范（定义如何创建规范的规范）

Philosopher 的本质是规则制定者。它不直接执行操作或设计系统，而是建立原则和约束，这些原则指导其他 Skill 的行为。Philosopher 的产出物是"宪法"——不可违反的根基性规则。

---

## SKILL.md 结构指导

### 保留在 SKILL.md 中的内容

- **Constitutional Rules**：宪法式规则（不可违反的核心原则）
- **Workflow**：基于原则的执行工作流（原则如何转化为行动）
- 两者必须明确分离，不混为一体

### 推入 `references/` 目录的内容

- 原则详解（每条原则的详细解释、背景、边界）
- 案例分析（原则的正面/反面应用案例）
- 推理模板（将原则应用于具体场景的模板）

```
skill-name/
├── SKILL.md          # 宪法规则 + 工作流（明确分离）
└── references/
    ├── principles/   # 原则详解
    ├── cases/        # 案例分析
    └── templates/    # 推理/应用模板
```

## 关键设计原则

### 1. 原则层与执行层分离

SKILL.md 中必须将宪法规则（Constitution）和执行工作流（Workflow）明确分为两个独立的部分。原则告诉 Agent "什么可以做、什么不可以做"，工作流告诉 Agent "如何在原则约束下完成任务"。

```
# SKILL.md 结构示例

## Constitution（宪法）
- Rule 1: ...
- Rule 2: ...
- Rule 3: ...

## Workflow（工作流）
- Step 1: ... (受 Rule 1 约束)
- Step 2: ... (受 Rule 2, 3 约束)
```

### 2. 宪法式约束（Constitutional Constraints）

原则必须是明确的、可判定的约束，而非模糊的指导方针。每条原则应该能回答"在具体场景 X 中，我应该怎么做"。

- 好的原则：「禁止在代码中硬编码 API key，必须使用环境变量」
- 差的原则：「注意代码安全性」

### 3. 递进式原则（Layered Principles）

原则应该分层组织，从最基础的不可违反规则，到推荐的最佳实践：

```
Layer 1: 绝对禁止（违反即失败）
Layer 2: 强制要求（除非有明确豁免理由）
Layer 3: 推荐实践（建议遵循，允许例外）
```

### 4. 原则的可测试性

每条原则都应该附带验证方法，Agent 可以自检是否违反了原则。

## 反模式警告

### 1. 原则和执行混为一谈

**症状**：宪法规则散落在工作流步骤中，无法一眼看到所有约束。
**后果**：Agent 可能在执行流程中忽略某些约束，因为它们被淹没在步骤描述中。
**修复**：将所有原则提取到 Constitution 部分，工作流中通过引用方式关联原则。

### 2. 原则过于抽象

**症状**：原则表述模糊（如"保持代码质量"），无法转化为具体行动。
**后果**：Agent 无法判断是否违反了原则，原则形同虚设。
**修复**：每条原则必须具体到可判定（是/否），并附带至少一个正例和一个反例。

### 3. 原则没有层次

**症状**：所有原则同等重要，没有优先级。
**后果**：当原则之间产生冲突时，Agent 无法做出取舍。
**修复**：将原则分为绝对禁止、强制要求、推荐实践三个层次，高层原则覆盖低层。

## 典型代表

### skill-creator-pro（Skill 创建器 Pro）

- Constitution: 定义了 Skill 创建的宪法规则
  - 结构规则：SKILL.md 必须包含什么、不能包含什么
  - 范式规则：如何选择和应用范式
  - 质量规则：什么样的 Skill 是合格的
- Workflow: 在宪法约束下的创建流程
- references/ 包含范式详解（即本目录）和评估标准
- 典型 Philosopher：宪法 + 工作流分离，原则层级清晰

### reasoning-enhancer（推理增强器）

- Constitution: 定义了推理质量标准
  - 禁止过早收敛
  - 禁止跳步推理
  - 禁止隐藏不确定性
- Workflow: 在推理约束下的问题分解和验证流程
- references/ 包含推理模板和案例分析
- 递进式原则：从"绝对禁止"到"推荐实践"

## 与相邻范式的区别

### Philosopher vs Architect

| 维度 | Philosopher | Architect |
|------|-------------|-----------|
| 抽象层级 | **原则/宪法**层 | **系统结构**层 |
| 产出物 | 规则、约束、标准 | 模板、框架、规范 |
| 控制方式 | 约束边界（不可做什么） | 定义结构（应该怎么做） |
| 复用方式 | 被其他 Skill 引用为约束 | 被其他 Skill 使用为模板 |

### Philosopher vs Navigator

| 维度 | Philosopher | Navigator |
|------|-------------|-----------|
| 知识类型 | 原则性知识（如何判断） | 事实性知识（在哪里找） |
| 使用方式 | 约束 Agent 行为 | 指引 Agent 查找 |
| 结构 | Constitution + Workflow | Decision Tree + References |
